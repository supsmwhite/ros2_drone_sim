# AI 辅助开发说明

## 1. AI 工具及其使用方式

本项目采用 ChatGPT 和 Codex 作为辅助开发工具。ChatGPT 主要用于考核要求分析、工程路线设计、技术问题讨论、Codex 任务编排以及实验与文档方案审查；Codex 主要用于仓库读取、代码实现与重构、测试补充、构建运行和问题定位。AI 参与覆盖动力学、控制器、任务管理、规划避障、自动评测和文档整理等环节。

ChatGPT 和 Codex 不作为项目的自主决策主体。每项 AI 任务均由开发者给出目标、允许修改范围、固定技术约定和验收条件；AI 产生的代码、参数建议和实验结论必须经过源码审查、自动测试、ROS2 闭环运行、数据分析和人工观察，才能进入最终版本。对于局部指标改善但整体收益不足、物理逻辑不一致或验证证据不完整的方案，开发者决定继续修改或回退。

## 2. 人机协作模式与责任划分

项目采用“开发者定义和决策，ChatGPT 分析与编排，Codex 实现与测试”的协作模式。开发者首先解析考核要求并确定初始系统边界。随着任务管理和静态规划功能加入，项目逐步形成最终六个 ROS2 软件包的职责划分及阶段优先级。在此基础上，把单一、可验收的任务交给 AI，并明确不能改变的坐标系、电机编号、Topic、Service、消息字段和物理单位。开发者还负责审查动力学、控制器、Mixer 和规划安全逻辑，设计测试与参数对比实验，在本机运行 ROS2 仿真和 RViz，并解释正式结果。

ChatGPT 主要协助把复杂目标拆解为阶段任务，讨论故障原因和实验设计，并为 Codex 形成包含实现边界、公式、接口、测试和交付要求的指令。Codex 根据这些约束完成局部实现、重构和测试补充，运行构建、测试与评测命令，并提供可供审查的修改结果。阶段是否完成、参数是否采用、实验是否有效以及结果能否进入报告，均由开发者依据工程证据作出决定。

| 工作环节 | 开发者职责 | AI 辅助内容 |
|---|---|---|
| 需求与架构 | 解析题目，确定功能边界、包和节点职责，安排阶段顺序 | ChatGPT 辅助拆解要求、识别依赖和风险 |
| 技术约定 | 固定 ENU/FLU、X 构型、电机顺序、单位和 ROS2 接口 | ChatGPT/Codex 检查约定在代码与文档中的一致性 |
| 算法与代码 | 审查公式、符号、异常处理和模块边界，决定修改是否保留 | Codex 草拟 C++、Python、Launch、YAML 并执行重构 |
| 测试与实验 | 设计验收场景和判断标准，运行仿真，观察 RViz，解释数据 | AI 补充单元/集成测试、评测脚本和参数对比方案 |
| 结果与取舍 | 综合安全、跟踪、时间、饱和和可复现性决定采用或回退 | AI 汇总日志、CSV、曲线和指标，辅助定位异常 |
| 文档与交付 | 核对最终事实、运行入口、能力边界和正式证据 | ChatGPT/Codex 辅助整理 README、AI_CONTEXT、报告和证据 |

## 四、推进原则

项目不按固定天数划分，而按能够独立验收的目标推进。

推荐总体顺序：

1. 工程结构与基础通信；
2. 动力学模型与独立测试；
3. RViz2 基础可视化；
4. 姿态与高度控制；
5. 三维位置控制；
6. 单目标点和多目标点；
7. 障碍物地图；
8. 碰撞检测和路径规划；
9. 静态避障与安全距离验证；
10. 稳定性测试、自动评测和加分功能；
11. 报告、README 和演示整理。

前一阶段没有稳定通过，不要大规模推进依赖它的后续阶段。

每一阶段都应遵循：

分析目标
→ 明确接口与物理原理
→ 给 Codex 一个范围明确的任务
→ 编译
→ 运行
→ 按验收场景测试
→ 检查真实结果
→ 修正问题
→ 更新文档
→ Git 提交和推送
→ 进入下一阶段

## 六、代码审查原则

Codex 完成工作后，你需要根据 GitHub 最新代码进行检查。

优先检查：

1. 数学公式是否正确；
2. ENU/FLU 坐标系是否一致；
3. 电机编号和旋转方向是否一致；
4. roll、pitch、yaw 力矩符号是否一致；
5. RPM、rad/s、力和力矩单位是否正确；
6. 消息中的参考坐标系是否正确；
7. Odom、IMU、TF 和 Path 是否一致；
8. 控制器和动力学接口是否匹配；
9. Mixer 是否与动力学力矩矩阵互为对应关系；
10. 参数是否合理且可配置；
11. 是否存在 NaN、发散、饱和和积分累积问题；
12. 代码是否有明确的测试；
13. 测试是否真正覆盖老师的验收要求。

当运行结果只有我本地能够获得时，请给我最直接的测试命令，并告诉我需要返回哪些关键输出。不要一次让我执行过多无关命令。

## 七、固定技术约定

除非实际代码已经明确修改并记录，否则继续遵循：

- 世界坐标系：ENU；
- x 向前，y 向左，z 向上；
- base_link：FLU；
- 重力沿世界坐标系负 z；
- X 型四旋翼；
- M1：前左，CCW；
- M2：后左，CW；
- M3：后右，CCW；
- M4：前右，CW；
- ROS2 接口使用 RPM；
- 动力学内部转换为 rad/s；
- 所有物理量使用 SI 单位；
- RViz2 负责显示，不负责计算动力学；
- 核心算法与 ROS2 节点通信代码分离；
- Launch、参数、URDF 和 RViz 配置集中在 drone_bringup。

如发现仓库真实代码与这些约定不同，请先指出差异，不要直接假设其中一方正确。

#### 交互过程与工程决策

该 Prompt 用作后续 ChatGPT 会话接管项目时的统一模板，实际使用时结合当时的考核题目、仓库地址和阶段状态补充项目上下文。模板将 AI 的参与方式限定为分阶段、可审查的工程协作，而不是一次性生成完整系统。其中将总体闭环拆分为工程结构、动力学、控制、任务、静态环境、规划、评测和交付，并明确只有前一阶段通过编译、运行和针对性验证后，才推进依赖模块。这一规则避免在动力学或姿态控制尚不稳定时提前叠加地图、规划和复杂功能，使故障能够在较小范围内定位。

每次 Codex 任务还必须包含目标、边界、固定约定、预期文件、公式、ROS2 接口、测试、预期结果和汇报要求。ChatGPT 据此协助形成阶段任务，Codex 负责局部实现和测试。人工审查持续覆盖数学公式、ENU/FLU、M1～M4 顺序、RPM/rad/s 单位、消息坐标系、Mixer 对应关系、NaN 与饱和保护，阶段切换和功能取舍始终由开发者决定。

#### 实现结果与验证

工程开发顺序与该方法一致：初始化后依次完成动力学、姿态/高度/水平控制、任务管理、静态环境、三维规划、安全连续轨迹、交互导航、扰动和评测。各阶段同时建立纯算法测试与 ROS2 集成验证，最终再由三个分层脚本和七组正式实验收束。最终工程结构以六个软件包的实际源码、公开 Launch 入口和正式测试与实验证据为准。

#### 交互过程与工程决策

开发者在项目初期即考虑到长期 AI 协作可能出现上下文漂移：如果目标、接口、已验证状态和当前问题只存在于对话中，新的 ChatGPT 会话或 Codex 任务容易沿用过时假设。为此，开发者要求先设计 README 和 AI_CONTEXT 的职责，而不是等代码规模扩大后再补文档。ChatGPT 提出的初步框架将 README 用于项目目标、结构、运行方式和稳定结果，将 AI_CONTEXT 用于阶段状态、技术约定、接口、验证和下一任务。

开发者没有机械保留最初建议的所有目录和计划内容，而是建立了明确的文档准入规则：代码生成不等于编译通过，编译通过也不等于功能验证；只有经过运行和验收的能力才能写入稳定状态。随着工程发展，早期计划中的包名、Launch 和文档路径被实际六包结构与三个公开入口替代，失效状态被覆盖而不是无限累积。

#### 实现结果与验证

最终 README 面向使用者，集中说明能力、入口、实验结果和边界；AI_CONTEXT 面向后续开发，记录当前包/节点职责、数据流、参数、正式协议和测试状态。两份文档在阶段收尾时均与源码和测试交叉核对，避免将骨架、试验分支或已弃用的早期窄通道导航试验场景误写为当前正式能力。这套分工也为 Codex 在每次任务前快速恢复可靠上下文提供了基础。

### 3.6 姿态闭环振荡定位与修正（交互摘要）

> **记录说明：** 3.6～3.11 未保留完整逐字 Prompt，以下内容根据代码修改、测试结果、正式实验和开发记录整理为交互摘要，不作为原始 Prompt 的逐字复现。

#### 问题与目标

### 3.8 连续轨迹切角问题与安全回退（交互摘要）

#### 问题与目标

A* 原始路径和简化折线通过碰撞检查后，五次连续轨迹仍可能在急转弯内侧偏离折线并进入膨胀障碍。RViz 中的曲线形状、轨迹碰撞复检和局部净空数据共同表明，离散路径的安全验证不能覆盖连续参考轨迹。本阶段目标是把安全检查扩展到控制器实际接收的连续轨迹。

#### 交互过程与技术判断

Codex 的初始轨迹方案以简化 waypoint 构造具有非零中间速度的五次曲线。开发者对比 planned、simplified 和 reference 三层几何后，定位到中间速度为了维持连续性而造成切角。仅检查有限个采样点仍可能遗漏两个采样时刻之间穿过障碍的情况，因此开发者要求同时检查每个采样位置和相邻采样位置之间的完整线段，并继续使用闭合 AABB 相交语义处理切触边界。

对于碰撞候选，开发者没有采用全局取消平滑或任意偏移 waypoint 的做法，而是要求 `PlannedTrajectoryBuilder` 保留原始 A* 索引，在碰撞轨迹段附近确定性回插原始路径点并重新构建。若几何安全但速度或加速度超限，则优先放大分段时间；如果所有速度比例、时间比例和局部细化候选仍不安全，执行器必须拒绝轨迹并进入安全保持。

#### 修改结果与验证

`test_planned_trajectory_builder.cpp` 增加 synthetic corner cut、远端目标原始点恢复、动态约束时间放大和无安全候选回退等测试。ROS2 集成测试再检查实际 Odom 相邻线段、最终保持和任务状态；RViz 用于观察绕行关系，但不替代自动碰撞判断。正式单目标避障和四目标导航均保持正安全净空，自动碰撞检查未检测到碰撞。测试和正式实验结果支持最终参考轨迹与实际飞行轨迹已经过完整安全链路检查。

### 3.9 多目标完成判定与航向规划（交互摘要）

#### 问题与目标

多目标任务如果只用位置距离判断完成，可能在车辆仍有较大速度或终端 yaw 尚未稳定时提前切换目标，造成快速切换和航向突变。开发者因此将完成条件扩展为位置误差、线速度、最短角 yaw 误差、角速度和持续保持时间同时满足，并要求多目标严格按顺序推进。

#### 交互过程与技术判断

Codex 实现共享的 `GoalCompletionGate`，开发者审查 `±π` 附近 yaw 误差必须使用最短角，并确认任何条件离开容差都会重置保持计时。对于静态导航航向，固定 yaw 无法自然表达沿路径转向；`path_tangent` 模式则需要在水平速度足够时使用轨迹切线，在垂直起飞或低速停稳时保持最近有效 yaw，在接近目标时平滑混合到 waypoint 终端 yaw，并限制每步变化率。

为避免单目标、多目标和 RViz 交互链出现三套不同实现，Codex 新增无 ROS 依赖的 `YawReferenceGenerator`。开发者要求它统一处理 Odom yaw 初始化、低速保持、`±π` 连续展开、一阶滤波、终端混合、最大 yaw 速率和非有限输入回退。接口审查确认 `TrajectorySetpoint` 只有 header、position、velocity、acceleration 和 yaw，没有 yaw-rate 字段，因此没有接受通过扩展消息来绕过当前控制接口的方案。

#### 修改结果与验证

单元测试覆盖基本切线方向、低速/垂直段、双向跨越 `±π`、终端混合、滤波、速率限制和非有限输入。完成门控测试覆盖 yaw 与角速度容差及保持时间。真实 ROS2 场景验证单目标、多目标和交互执行复用相同算法；正式四目标导航按 `[0,1,2,3]` 顺序完成，每个目标均具有独立终端 yaw，未发生目标跳过或 waypoint 切换时的航向突跳。

### 3.10 导航参数实验与方案取舍（交互摘要）

#### 问题与目标

稳定基线已经能够完成全地图导航，本项目进一步希望在不破坏安全和控制余量的前提下改善任务时间。速度与加速度参数候选方案 A2 的采用评估被限定为只调整 `nominal_speed`、`max_reference_speed` 和 `max_reference_acceleration` 及其 Launch 透传，不同时修改控制器、A*、地图、安全半径或轨迹算法；原保守参数作为命令行 override 保留，用于对照和回退。

#### 交互过程与技术判断

ChatGPT 协助制定单变量实验和停止条件，Codex 执行配置迁移与测试。对比表明，只提高期望速度会受到连续轨迹加速度验证和时间放大的限制；最终 A2 使用 `0.50 m/s`、`0.90 m/s` 和 `0.60 m/s²`，在重复完整地图运行中改善任务时间，同时保持无碰撞、无非有限值和可接受跟踪误差。正式采用时从稳定主线建立干净分支，只迁移正式配置、Launch 参数透传、结构测试和必要文档，不将扫描工具或试验算法带入主线。

路径简化实验进一步比较了原始简化基线 S0（推荐净空 `0.00 m`），以及推荐净空分别设为 `0.18 m`、`0.20 m` 和 `0.25 m` 的 C18、C20、C25 三组候选方案。C18/C20 提高了两个指定低净空区域，但简化后的 waypoint 增加、任务约慢 `5.3 s`，并使其他拐角的局部净空下降；C20 相比 C18 没有额外收益，按预设停止条件不再运行 C25。该结果说明局部几何改善不能替代全局时间、净空、路径形状和控制余量的综合判断。

#### 修改结果与验证

A2 经 fast、assessment、full 回归及重复导航验证后进入正式配置，参数为 `nominal_speed=0.50`、`max_reference_speed=0.90`、`max_reference_acceleration=0.60`。推荐净空逻辑未被采用；试验中的 `shortcut_preferred_clearance=0.0` 表示 S0 基线和原有简化行为。这一取舍保留了经重复运行验证的性能收益，避免以单项局部最优换取整体退化。

### 3.11 扰动实验与评测指标修正（交互摘要）

#### 问题与目标

扰动扩展用于验证外力下的位置保持与撤力恢复。实验设计包含 Short Gust 和 Persistent Release 两类场景：前者观察短时外力造成的瞬态偏移和恢复，后者观察持续偏置下受限水平积分的补偿及撤力后的卸载。外力被限定为作用于质心的 `map` 系等效力，不描述为完整风场。

#### 交互过程与技术判断

Codex 增加 `/drone/external_wrench`、输入限幅和超时清除，并实现水平受限积分、捕获范围、积分限幅、saturation back-calculation 与反向卸载。开发者通过外力方向、位置误差、积分补偿和 diagnostics 联合观察，确认积分用于降低持续偏置，同时不能在饱和或误差反向后无限累积。两类扰动分别运行，避免把持续补偿与短时瞬态混为同一指标。

Recorder、Analyzer、CSV、事件、`summary.json`、metadata、参数快照、校验和及 manifest 构成正式证据链。对 CSV 和阶段事件交叉检查后发现，原 tracking 统计从任务启动开始，地面起飞造成的高度差会把 full-mission 最大误差推近起飞高度，不能代表规划轨迹跟踪。因此分析器增加 navigation phase 起点，正式导航报告采用 `navigation_tracking_max_error_m` 和 `navigation_tracking_rms_error_m`，同时保留 full-mission 指标但不混用。

恢复指标也经过语义修正。首次进入位置/速度门限只能说明瞬时满足条件，Recorder 还要求在保持窗口内持续满足；因此正式报告区分 `recovery_threshold_entry_time_s` 和 `recovery_confirmed_time_s`，并采用后者说明恢复完成。Persistent Release 撤力时已处于门限内，约 `1.005 s` 表示保持后确认，而不是重新进入门限所需时间。

#### 修改结果与验证

Short Gust 与 Persistent Release 均生成独立正式记录，通过自动分析和人工曲线/RViz 验收。导航与扰动指标定义被写入分析代码、结果说明和报告指标生成流程；参数快照和校验和用于防止结果与运行配置脱节。七组正式实验均通过自动分析、证据完整性检查和人工验收，具备正式报告引用资格。各组实验保留其实际运行时的代码提交和参数快照，不将它们统一归属于后续的工程状态锁定版本。

## 5. 开发者审查和修改的核心公式与接口

### 5.1 电机响应与四旋翼动力学

ROS2 接口以 RPM 表示电机指令，动力学内部以 rad/s 计算。Codex 初始实现包含单位换算、一阶电机响应和推力平方模型：

```text
Ω = 2πn / 60
Ω(k+1) = Ω(k) + (1-exp(-dt/τ_m))(Ω_cmd-Ω(k))
F_i = k_F Ω_i²
Q_i = k_M Ω_i²
```

开发者确认一阶离散式使用当前实际转速而非直接把命令当作瞬时转速，并检查 RPM 限幅与非有限值处理发生在状态积分之前。电机顺序固定为 M1 前左 CCW、M2 后左 CW、M3 后右 CCW、M4 前右 CW。令有效力臂 `ℓ=L/√2`，X 构型合力与力矩为：

```text
T  = F1 + F2 + F3 + F4
τx = ℓ(F1 + F2 - F3 - F4)
τy = ℓ(-F1 + F2 + F3 - F4)
τz = -Q1 + Q2 - Q3 + Q4
```

开发者通过正负 roll、pitch、yaw 独立输入核对符号，而不是只检查对称悬停。总推力沿 `base_link` 的 `+z`，世界重力沿 `map` 的 `-z`。包含外力、阻力和角阻尼的刚体关系为：

```text
p_dot = v
v_dot = (R(q)F_body + F_external + F_drag)/m + g_world
Iω_dot = τ_rotor + τ_external - D_ωω - ω×(Iω)
```

`q` 表示 body-to-world，机体系角速度产生的增量四元数右乘并在每步归一化。正式节点只接受 `map` 系质心外力，外部力矩配置上限为零。模型测试和姿态闭环联合测试共同验证力矩矩阵、姿态更新与控制链方向一致。

### 5.2 位置、高度与姿态控制

水平控制使用世界系位置/速度误差、轨迹加速度前馈和受限积分：

```text
a_xy = Kp(p_d-p) + Kd(v_d-v) + a_ff + a_integral
```

Codex 实现水平加速度限制、最大倾角限制、积分捕获范围、积分模长限制、saturation back-calculation 和误差反向时的 integrator unloading。开发者通过正负 x/y 误差和速度反馈测试确认控制方向，通过持续外力和撤力实验检查积分既能补偿稳态偏置，又不会在饱和或误差反向后继续累积。

高度控制为 PD 加前馈，不含高度积分：

```text
a_z = Kp(z_d-z) + Kd(v_zd-v_z) + a_z,ff
F_z = m(g+a_z)
T = F_z / cos(tilt)
```

开发者核对重力补偿、当前姿态的倾角余弦、倒置姿态拒绝、最小余弦保护和总推力限幅。姿态控制使用误差四元数最短旋转表示：

```text
q_e = q⁻¹⊗q_d
e_R = 2 vec(q_e)
τ = K_R e_R + K_ω(ω_d-ω)
```

开发者审查 `q/-q` 双重覆盖、精确半圈确定性符号、角速度 D 项方向和三轴力矩限幅，并通过完整动态链测试修正 roll/pitch 阻尼。整套控制系统不能称为完整 PID：水平环为 P+D+受限 I+前馈，高度和姿态环为 PD。

### 5.3 Mixer 与电机指令

Mixer 使用与动力学完全相同的 M1～M4 顺序和 X 构型符号。Codex 实现解析逆映射，将总推力和三轴力矩转换为四个单电机推力，再由

```text
Ω_i = sqrt(max(F_i,0)/k_F)
n_i = 60Ω_i/(2π)
```

得到 RPM。开发者用纯 roll、pitch、yaw 和混合 wrench round-trip 测试确认 Mixer 与动力学矩阵互为对应；负单电机推力先钳零，随后执行 RPM 上下限，并通过 `ControllerDiagnostics` 区分整体饱和和逐电机裁剪。非有限输入返回安全零命令。正式结果中的 RPM 是 `/drone/motor_rpm_cmd` 上的控制器指令，不是电机实测转速。

### 5.4 规划与连续轨迹安全

开发者确认 A* 使用三维 26 邻域，累计代价和启发函数均为欧氏距离；原始 AABB 按 `safety_radius + planning_margin` 膨胀，节点和连接边同时检查碰撞。`PathSimplifier` 删除中间点前必须重新检查跨点直线，不能继承原路径的安全结论。

分段五次轨迹除有限性、端点、速度和加速度外，还检查每个采样点及相邻采样点间的线段。碰撞时按原始 A* 索引局部回插点，纯动态约束失败时放大分段时间；无安全候选则拒绝执行。正式安全净空定义为质点到原始 AABB 的欧氏距离减 `safety_radius`，不是对已膨胀障碍再次扣减半径。

### 5.5 ROS2 Topic、Service 与 TF

开发者逐项核对实际数据流中的主接口：`/drone/goal`、`/drone/trajectory_setpoint`、`/drone/motor_rpm_cmd`、`/drone/odom`、`/drone/imu`、`/drone/path`、`/drone/external_wrench`、`/drone/controller/diagnostics` 和 `map → base_link` TF。

`MotorRPM` 的四个字段显式对应 M1～M4；`TrajectorySetpoint` 包含 header、position、velocity、acceleration 和 yaw，不包含 yaw-rate。`ExecuteGoalSequence` 请求为 `geometry_msgs/PoseArray goals` 与 `uint64 draft_revision`，响应为 `bool accepted` 和 `string message`。`waypoint_manager_node` 提供 `/drone/mission/execute`，`multi_goal_static_avoidance_node` 提供 `/drone/interactive_goals/execute`。Odom pose 在 `map`，child 为 `base_link`，twist 随 child frame 在 `base_link` 表达；TF 与 Odom 使用同一位置和姿态。

| 离散路径安全未覆盖连续轨迹 | 初始实现只验证 A* 路径与简化折线，未覆盖五次轨迹在急转弯处的切角。 | RViz 几何关系、连续轨迹碰撞复检和局部净空数据暴露了检查对象不完整。 | 增加轨迹采样点及相邻线段检查，并加入局部回插、时间放大和无安全候选时拒绝执行。 | `PlannedTrajectoryBuilder` 单元测试、ROS2 集成测试和两组正式导航均未触发碰撞检查。 |
| 导航误差包含起飞阶段 | 全任务 actual-to-reference 误差将地面起飞与正式导航合并统计，不能直接表示规划轨迹跟踪性能。 | CSV、轨迹曲线与路径事件时间对齐显示，最大误差受起飞高度差主导。 | 分离 full-mission 与 navigation phase，正式导航跟踪采用 navigation phase 的最大值和 RMS。 | Analyzer 测试与正式导航结果采用一致的阶段定义。 |
| 恢复时间语义不清 | 首次进入位置/速度阈值容易被误解为已经完成稳定恢复。 | Recorder 状态机、`events.csv` 和 `summary.json` 表明，任务完成还要求持续满足保持窗口。 | 分别记录 threshold entry 和 confirmed recovery，正式报告使用 `recovery_confirmed_time_s`。 | Recorder、Analyzer、结果说明和报告指标生成流程采用同一恢复判据，Persistent Release 的约 `1.005 s` 明确表示恢复确认时间。 |

上述案例表明，AI 输出的主要风险并非只来自语法或编译错误，也可能来自控制闭环范围、安全验证对象和指标语义的不完整。项目通过源码审查、对照实验、自动测试和正式运行对这些问题进行了修正。

## 7. 动力学、控制器与 ROS2 接口验证

### 7.1 动力学验证

动力学首先由 `test_quadrotor_model.cpp` 进行无 ROS 依赖的纯模型验证。测试覆盖零 RPM 自由落体与地面约束、对称推力、roll/pitch/yaw 力矩方向、电机一阶响应、RPM 限幅、外力、阻力/角阻尼、非有限输入和姿态有效性。判断标准不仅是状态有限，还包括三轴符号、非指令轴力矩接近零、四元数归一化及 IMU specific force 的物理含义。

`test_attitude_closed_loop.cpp` 进一步串联姿态控制、Mixer、电机响应和动力学，验证正负 roll/pitch 与水平姿态在 20 s 内稳定。节点运行时检查 Odom、IMU、Path 和 TF 的 frame、姿态与频率。正式 Hover 最终位置误差约 `0.000334 m`、最终速度约 `0.000436 m/s`，无姿态发散且无饱和样本；RViz 人工观察同时覆盖起飞和悬停姿态。

### 7.2 控制器验证

控制器 GTest 分别覆盖 `MotorMixer`、`AttitudeController`、`AltitudeController`、`HorizontalPositionController`、`HoverController` 和组合 `PositionController`。测试检查正负误差方向、速度阻尼、重力与倾斜补偿、姿态最短旋转、Mixer round-trip、前馈、积分限幅、back-calculation、unloading、非有限保护和各级饱和传播。

ROS2 闭环场景进一步验证单目标、基础四目标、轨迹跟踪和扰动撤除。根据 `/drone/controller/diagnostics` 区分瞬态与终态饱和：Single Goal 有 `21` 个瞬态饱和样本，Basic Multi-goal Mission 有 `202` 个，但二者结束时均未饱和，因此不将结果表述为“全过程无饱和”。两组正式导航和两组扰动的饱和样本为 0。

### 7.3 规划与避障验证

规划层的纯算法测试覆盖 AABB 膨胀、点/线段碰撞、三维 A*、确定性简化、安全连续轨迹、yaw 生成和完成门控。判断标准包括路径存在性、所有节点与边安全、简化索引严格递增、轨迹端点一致、速度/加速度不超限、采样点与相邻线段不碰撞，以及失败时返回明确状态。

ROS2 集成场景验证规划路径、简化路径、连续参考和实际轨迹在同一执行链中的关系。正式单目标静态避障的 navigation tracking 最大误差约 `0.02342 m`、最小安全净空约 `0.15018 m`；四目标三维导航完成 4/4 个目标，navigation tracking 最大误差约 `0.03263 m`、最小安全净空约 `0.18152 m`。两者均无碰撞。RViz 用于人工检查障碍、目标、路径和高度变化，自动净空与碰撞结果则作为安全结论的量化依据。

### 7.4 ROS2 Topic、Service 与 TF 验证

工程初始化和阶段验收使用 `ros2 node list`、`ros2 topic list`、`ros2 topic info/echo/hz`、`ros2 interface show`、`ros2 service list/type` 以及 `ros2 run tf2_ros tf2_echo map base_link`。这些检查用于确认节点实际存在、发布者/订阅者类型匹配、消息字段正确、Odom 更新频率稳定和 TF 可查询。

Launch 集成测试继续验证 `/drone/mission/execute` 与 `/drone/interactive_goals/execute` 的接受/拒绝行为、目标顺序、revision、预检失败安全、Odom 新鲜度、外力超时和 diagnostics。接口命令的输出还与源码中的节点归属、消息定义和 Launch 装配进行交叉核对，避免仅因 Topic 名称存在就认定数据语义正确。

### 7.5 正式实验与证据管理

分层测试入口为 `bash scripts/test_fast.sh`、`bash scripts/test_assessment.sh` 和 `bash scripts/test_full.sh`。最终回归结果分别为 fast `30 CTest / 313` 内部用例、assessment `5 / 16`、full `34 / 326`，均为 0 errors、0 failures、0 skipped。快速层覆盖工具、核心算法和选定节点，assessment 层覆盖正式入口，full 层覆盖整个工作空间。

正式场景包括 Hover、Single Goal、Basic Multi-goal Mission、Full-map Static Avoidance、Multi-goal 3D Navigation、Short Gust 和 Persistent Release。Recorder 保存样本、路径与事件，Analyzer 生成 `summary.json` 和图表；metadata、参数快照、SHA-256 和 `results/manifest.json` 记录运行来源与证据完整性。正式实验由开发者在本机实际执行，并结合 RViz、曲线、CSV 和 summary 完成最终人工验收。七组结果由各自实际运行时的历史提交生成，`results/manifest.json` 保留了对应关系，因此不将它们误述为全部由同一后续版本生成。

## 8. 总结

ChatGPT 和 Codex 深度参与了需求分析辅助、阶段任务设计、代码草拟与重构、测试补充、问题排查、自动评测和文档整理。其参与范围覆盖六个 ROS2 软件包及正式实验工作流，而不是局限于文字建议。AI 提高了代码阅读、方案比较和重复性工作的效率，也在姿态振荡分析、连续轨迹安全和指标语义方面暴露出局部判断或验证范围不足的问题。

开发者在整个过程中负责定义任务、限制修改范围、固定坐标与接口、审查动力学和控制公式、设计对照测试、运行仿真、解释数据并决定采用或回退。项目的最终结论由纯算法测试、动力学—控制器联合测试、ROS2 Launch 集成测试、Topic/Service/TF 检查、正式数据证据和 RViz 人工验收共同支持，任何单一 AI 输出都不能替代这些验证。

系统边界保持明确：静态障碍仅用于规划与碰撞监测，不产生物理接触反作用；外力为作用于质心的集中等效力，不是完整风场；系统未实现动态障碍、局部重规划、MPC 或完整姿态规划。这些边界同样由开发者根据考核目标、稳定性和可验证性作出取舍。
